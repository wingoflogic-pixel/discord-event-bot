# 管理画面リデザイン（Discord API ピッカーで手打ち廃止・RRULE 非表示・2 層シェル）

オープンソースとして公開できる品質へ管理画面を作り直すにあたり、UI レベルで後戻りしにくい I/F 判断を行う。背景・用語は [CONTEXT.md](../../CONTEXT.md)、上位のモデル判断は ADR [0004](./0004-multi-server.md)／[0005](./0005-drop-event-layer.md)／[0006](./0006-member-three-layers.md)。

## 決定

1. **2 層シェル**：ログイン後にまず全画面で**サーバーを選択** → 左サイドナビのワークスペースへ。ヘッダに現在サーバーと「サーバー切替」を常設。選択中サーバーは URL ハッシュ＋localStorage で保持。セクションは **通知 / メンバー区分 / 回答履歴**（旧「記録」を改称）。サーバー内ナビにグローバルなメンバー一覧は置かない。
2. **Snowflake 手打ちの全廃**：`guild_id`・User ID・`channel_id` のテキスト入力を、Discord API 由来のピッカーへ置換する。
   - サーバー：`GET /users/@me/guilds`（ADR 0004）
   - メンバー：`GET /guilds/:id/members`（ADR 0006・特権インテント要）
   - チャンネル：`GET /guilds/:id/channels`（テキストチャンネルを列挙・特権不要）
3. **繰り返し UI の平易化**：生 RRULE を一次 UI から隠し、「毎週 土曜日 21:00〜」等の人間可読サマリを表示。保存は RRULE 文字列のまま（ADR [0002](./0002-rrule-recurrence.md) 維持）。「基準日／anchor_date」はユーザー語彙から廃し、**隔週選択時のみ**「次回の開催日」候補を提示。種別／モードに応じた条件表示を徹底する。

## トレードオフ

- **チャンネルピッカーの限界**：bot がアクセスできないチャンネルは候補に出ず、権限不足チャンネルを選ぶと投稿時に失敗する。手打ちの自由度を捨てる代わりに事故と認知負荷を減らす。
- **新たな実行時依存**：各ピッカーが Discord API に依存する（可用性のクリティカルパス化は ADR 0004 のトレードオフに準じる）。
- **表現力の意図的な制限**：ビルダのプリセット（週次／隔週／毎月第N曜＋単発）しか UI から作れず、ADR 0002 が保存形式に確保した表現力（隔月・毎年等）へは UI から到達できない。プリセット外は将来のビルダ拡張に委ね、当面は非対応とする（ADR 0002 と相互参照）。
- 番号割り当て（Assignment）の実行は「回答履歴」から開催回（Occurrence）の文脈へ移すため、操作面の再配置が必要。
