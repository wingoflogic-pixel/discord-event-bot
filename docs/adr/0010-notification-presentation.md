# 通知の体裁（回答不要・メンション方法・カスタムメッセージ）

通知の「投稿の見え方」を 3 つの軸で運用者が制御できるようにする。**回答要否**（出欠ボタンを出すか）、**メンション方法**（ロール / バイネーム / なし）、**カスタムメッセージ**（見出し＋本文）。スケジュール（recurring/oneoff）とは独立した直交軸として扱う。

## 文脈

実運用には「基本出席が前提で出欠回答を取る必要がない」ケースがある（例: あるスタッフは第1・3・5日曜が必須出勤、別のスタッフは第2・4日曜が必須）。こうした対象には**告知だけ**を出したい。また告知の宛先は、ロール（`@staff`）一括ではなく**担当者をバイネームで個別に呼びたい**こともある。さらに告知文の見出し・本文は通知ごとに違う文面にしたい（どのトラックの告知か区別したい）。

従来は通知が常に出欠ボタン付きで投稿され、メンションは `mention_enabled`(0/1) のロールメンションのみ、文面は `📅 イベント募集開始!` 等のハードコードだった。

## 決定

- **回答要否は recurring 専用のフラグ `requires_response`（0/1・既定1）**。第3の `type`（`announce` 等）にはしない。スケジュール（recurring/oneoff）と回答要否は直交し、type を増やすと組合せが爆発するため。
  - `requires_response=0`（回答不要・通知のみ）: 募集投稿は**ボタンなしの告知 1 本**になり、Response を一切集めない。**未回答/未定リマインド・ノルマ・番号割り当ては対象外**（cron でも UI でも無効化）。再投稿防止のため occurrence 行は従来どおり生成する。
  - **oneoff は常に回答あり**（このフラグを持たせない）。単発・複数候補は本質的に「可否を集める」機能であり、回答不要と矛盾するため。
- **メンション方法は列挙 `mention_mode`（`none`/`role`/`members`）**。旧 `mention_enabled`(0/1) を置換し、migration で `1→role`,`0→none` に移行。「ロールかバイネームか」は排他選択なのでブール2本ではなく列挙1本にし、不正状態（両方オン）を作れないようにする。
  - `members`（バイネーム）: 区分の**アクティブ Member**（休止中除外）を `<@id>` で個別列挙。**メンションに割ける字数は「2000 −（見出し＋本文＋日時行の実長）」から動的に逆算**し、超過分は「**ほかN名**」と省略する。これにより合成後の content が必ず Discord の 2000 字上限に収まる（バイネームは少人数の区分前提）。`@everyone` 区分でバイネームを選ぶと全員個別メンションになるため UI で警告する。最終防衛として送信直前にも総長クランプを置く。
  - mention_mode は全通知（recurring/oneoff・回答要否を問わず）で選択可能。
  - **`allowed_mentions` で解析対象を mode に絞る**（none=何もping無し / role=ロール（@everyone は everyone）/ members=ユーザーのみ）。本文に書かれた `@everyone` 等で意図しない一斉メンションが飛ぶのを防ぎ、`none` の「メンションしない」保証を担保する。
- **カスタムメッセージは見出し（必須）＋本文（任意）の 2 フィールド**。
  - `message_title`: 必須・1 行・最大100字。投稿の 1 行目に `**太字**` で出る。**既定文言へのフォールバックは持たない**（運用者が必ず書く）。
  - `message_body`: 任意・複数行・最大1500字。空なら省略。
  - 合成は `{メンション}` → `**見出し**` → `本文?` → `日時:`（自動）→ ボタン（回答ありのみ）。**日時行とボタンはシステム自動**で、本文への日時埋め込み（置換変数）は持たない（日時は動的なので必ず自動行で出す）。
  - 適用先は**チャンネルへのメイン投稿**のみ: recurring の募集/告知・単発単一の募集・単発複数候補の**調整ヘッダ**（各候補スロットの可否投稿はシステム定型）。**DM（未回答/未定リマインド）・ノルマ DM・番号割り当ての公開投稿には適用しない**（機能的通知は定型のまま）。

## 根拠

- フラグ採用で「単発の告知だけ」等の将来の組合せにも type を増やさず対応できる。回答不要を recurring 限定にしたことで、複数候補調整との矛盾組合せ自体が発生せず実装・UI が単純になる。
- `mention_mode` 列挙は排他選択を型で表し、`mention_role_id`（ADR [0009](./0009-role-backed-segment-membership.md)）をメンバー源に兼用しているため `members` で「ロール保有者をバイネーム」も自然に表せる。
- 見出し必須・既定なしは「投稿の顔は運用者が決める」を徹底し、ハードコード文言の散在をなくす。日時/ボタンを自動に固定することで、動的情報の取りこぼし（日時未記載・ボタン忘れ）を防ぐ。

## トレードオフ

- バイネームは Discord の 2000 字上限に縛られ、大人数区分や長い本文との併用では「ほかN名」省略で**全員には個別通知が届かない**（合成後は動的予算で必ず上限内に収め投稿失敗＝無音化は防ぐが、省略自体は残る）。確実な全員通知が要るならロール／`@everyone` を使う。
- 見出しが必須になったため、**既存通知には見出しが無い** → migration で現行文言 `イベント募集開始!` にバックフィル（挙動維持・以後編集可能）。単発・複数候補の旧ヘッダ文言 `イベント候補日の調整!` も同値にバックフィルされる点は許容。
- `mention_enabled` 列は SQLite の列削除を避けるため**残置**（deprecated・本体は参照しない）。将来クリーンアップの余地は残す。
- 本文への置換変数（`{日時}` 等）が無いため、日時を文中の任意位置に差し込むことはできない（必ず自動の日時行に出る）。

## スキーマ影響（migration 0009）

`notifications` に 4 列追加:
- `requires_response INTEGER NOT NULL DEFAULT 1`（既存=回答あり維持）
- `mention_mode TEXT NOT NULL DEFAULT 'role'`（`mention_enabled` から移行）
- `message_title TEXT NOT NULL DEFAULT ''`（既存は `イベント募集開始!` でバックフィル）
- `message_body TEXT`（null 可）

`mention_enabled` は残置（deprecated）。
