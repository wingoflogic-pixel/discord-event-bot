# マルチサーバー対応（テナント分離はしない）

1 デプロイで**複数の Discord サーバー**を扱う。ADR [0003](./0003-generality-boundaries.md) が「次弾」へ送った `guild_id` 分離を前倒しで採用するが、**マルチテナント（認可境界）には踏み込まない**。

## 決定

- `segments` と `notifications`（Event 廃止後・ADR [0005](./0005-drop-event-layer.md)）は `guild_id`（Discord Snowflake）で 1 つのサーバーに属する。
- 管理画面は最上位で**サーバーを選択**し、以降の操作はそのサーバー配下に限定される（全画面サーバー選択 → 左サイドナビのワークスペース）。
- サーバー一覧と表示名は **Discord API（`GET /users/@me/guilds`＝bot の参加サーバー）から取得**し、DB に `guilds` マスタは持たない。作成フォームの `guild_id` は手打ちをやめ、実サーバーのドロップダウン選択にする。

## 非目標（ADR 0003 を踏襲）

- **テナント分離はしない**：サーバーごとのアクセス制御・認可境界・Bot 招待時オンボーディングは持たない。単一の運用者が単一の `ADMIN_TOKEN` で全サーバーを管理する。
- 言語＝日本語固定・時刻＝JST 固定は維持（ADR 0003 の他 2 境界は有効）。

## トレードオフ

- **可用性のクリティカルパス化**：管理 UI の最上位（サーバー選択）が Discord API のライブ取得に依存するため、API のレート制限・障害・トークン失効時はサーバー選択ができず管理 UI 全体が操作不能になりうる。短期キャッシュで緩和しつつ、フォールバックとして **DB 上に出現する distinct `guild_id` を退避選択肢**として提示する（名前は不明表示）。`GET /users/@me/guilds` は bot トークンのみで可（特権インテント不要）。
- bot が抜けたサーバーの `guild_id` を持つ行は「不明なサーバー」として一覧には出すが、新規作成は不可・既存の閲覧／移管のみ可とする。
- **ブラスト半径の増大**：認可は単一 `ADMIN_TOKEN` のままで、マルチサーバー化により**トークン漏洩時の影響が全参加サーバーへ拡大**する。自己ホスト・運用者単独の前提で許容するが、複数運用者の権限分離は将来課題。

Supersedes: ADR [0003](./0003-generality-boundaries.md) の「単一テナント」境界。
