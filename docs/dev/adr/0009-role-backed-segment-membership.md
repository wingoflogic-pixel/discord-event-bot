# ロール連携メンバー区分（Discord ロールを正本にメンバーを完全同期）

メンバー区分（Segment・ADR [0006](./0006-member-three-layers.md)）の所属を、運用者が手動で積み上げるのではなく、**Discord ロールを正本（source of truth）として完全同期**できるようにする。`mention_role_id` を「メンション先」と「メンバー源」に兼用する。

## 文脈

ADR 0006 では区分所属（③ 出欠対象）を「手動追加 or 募集ボタン回答時に自動生成」していた。実運用では「キャスト」「スタッフ」等の区分が Discord ロールと 1:1 で対応することが多く、ロール付け外しのたびに区分も手で直すのは二重管理・ドリフトの温床になる。Workers は Gateway を持たない（interactions＋cron のみ）ため**リアルタイム追従はできず**、プル型の同期が前提となる。

## 決定

- **正本＝Discord ロール／完全同期**：`mention_role_id` が設定された区分（ロールID または `@everyone`）は「ロール管理区分」とし、所属は**ロール保有者の集合に完全照合（不足を追加・余剰を除外）**する。ソース種別フラグは持たず、区分のロール有無だけで判定する。
- **兼用**：`mention_role_id` はメンション先とメンバー源を兼ねる。これにより「区分＝そのロールの保有者、メンションも同じロール」と意味が一貫する。代償として「ロールをメンションしつつメンバーは手動で一部だけ」はできない。
- **同期トリガ**：管理画面の「ロールから同期」ボタン（即時）＋**日次 cron 開始時**に、有効通知が参照するロール管理区分を自動同期してから募集/リマインド/ノルマ処理を行う。
- **回答ゲート**：ロール管理区分の出欠ボタンは、interaction に同梱される `member.roles` で押下者のロール保有を判定（追加 API 不要）。保有者／`@everyone` は回答可＋即メンバー化、非保有者は ephemeral で拒否。手動区分は従来どおりボタン回答で自動所属。
- **休止は別軸で保持**：`segment_members.status`（`''`/`休止中`）は「ロールは持つが一時的に対象外」というオーバーライド。同期は `addSegmentMember`（upsert・status 維持）と `removeSegmentMember` の差分照合で行い、ロールを保有し続ける限り休止は保持される。ロールを失えば除外され休止も消える（再付与時はアクティブで復帰）。
- **@everyone と bot**：`@everyone` は全人間メンバー（bot は常に除外）。完全同期し、取得は既存上限（20 ページ＝2万人）を維持。`@everyone` 設定時は「全員がリマインド/ノルマ DM の対象になる」と UI で警告する。
- **管理 UI**：ロール管理区分のメンバー一覧は読み取り専用（手動 追加/外す は非表示）＋「ロールから同期」ボタン＋最終同期時刻。**休止/再開（および `/pause` `/resume`）は引き続き可能**。
- **無人化・失敗の安全策**：メンバー取得が失敗（Server Members Intent 無効・一時エラー）したら**同期せず既存を保持**（手動同期はエラー表示）。有効レスポンスでロール保有者 0 人かつ現メンバーが残る場合、**cron は「全員除外になる同期」をスキップ＋警告ログ**（ロール削除・指定ミスでの無人化事故防止）。手動同期のみ確認ダイアログ経由で空にできる。
- **移行**：手動運用中の区分にロールを設定すると、初回同期も完全照合（非保有の現メンバーは除外）。破壊的なため保存時／同期時に確認ダイアログを出す。ロールを外す（null 化）と手動モードに戻り、その時点のメンバーは凍結＝手動編集可。

## 根拠

- 区分が Discord ロールと一致するのが一般的で、ロールを正本にすれば二重管理が消える。`@everyone` で「全員」も一発で表せる。
- `member.roles` が interaction に含まれるため、回答ゲートを**追加の Discord API なし**で実現でき、母集団がロールと常に一致する（押したのに集計に出ない/居ないのに残る、を防ぐ）。
- 回答（`responses`）・割り当て（`assignments`）は occurrence×user 主キーで所属と独立。除外しても履歴は消えない。

## トレードオフ

- ADR 0006 と同様、メンバー取得に **Server Members Intent が必須**。未有効だと同期できない（その場合は既存を保持してスキップ）。
- リアルタイム追従は不可（Gateway なし）。所属はあくまで「最後に同期した時点」のスナップショットで、手動同期と日次 cron の間は古くなりうる。
- `mention_role_id` 兼用により「メンション用ロール ≠ メンバー用ロール」は表現できない。必要になれば将来 `member_role_id` を分離する余地は残す。
- ロール管理区分では手動の細かな所属調整（特定 1 名だけ追加/除外）はできない（休止での一時除外は可能）。
- `@everyone` × 大規模サーバーは大量 DM・大量書き込みになりうる（小規模コミュニティ前提・UI 警告で緩和）。

## スキーマ影響

- `segments.members_synced_at TEXT`（最終同期時刻・null 可）を追加（migration 0008）。`mention_role_id` の意味は不変（値の解釈に「メンバー源」を追加するのみ）。
