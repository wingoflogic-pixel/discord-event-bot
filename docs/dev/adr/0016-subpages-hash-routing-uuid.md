# フォーム系モーダルを子ページへ移行し、ハッシュルーティングと UUID シャドウキーで 3 層シェルへ拡張する

ADR [0007](./0007-admin-ui-redesign.md) は「2 層シェル（サーバー選択 → 左サイドナビのワークスペース）」を定めたが、CRUD フォーム類がモーダル中心の構造になっており、(1) URL でディープリンクできない、(2) ブラウザバック／リロードで復帰できない、(3) 大きなフォーム（通知・グループ分け）に対して `.modal-lg` の幅と縦スクロールが窮屈、という問題が顕在化した。**フォーム系モーダルを子ページへ移行し、3 層シェルに拡張する**。

## 決定

1. **3 層シェル**: 既存の 2 層（ヘッダ＋左サイドナビ）の上に、子ページ層を追加する。子ページは**メインエリアのみ**を差し替え、ヘッダと左サイドナビは常駐する。サイドナビは子ページ表示中も親タブ（通知 / メンバー区分）をハイライトし続ける。

2. **ハッシュルーティング**: URL ハッシュを階層パスとして使う。形は `#g/<guildId>/<resource>/<uuid?>/<sub?>` で、`hashchange` イベントで小さな自前ルータを駆動する。`<guildId>` は Discord Snowflake のまま（ADR [0004](./0004-multi-server.md) 維持）。

3. **UUID シャドウキー**: URL／API に露出する次の **6 テーブル**に `uuid TEXT UNIQUE NOT NULL` を追加する。内部の `INTEGER PRIMARY KEY AUTOINCREMENT` は維持し、URL／API では UUID を使い、DB クエリ前に UUID → internal id へ 1 段 lookup する。
   - `notifications` / `segments` / `occurrences`
   - `groupings` / `groups` / `grouping_constraints`

   発番は `crypto.randomUUID()`（Workers ランタイムでネイティブ）。既存行はマイグレーション 0012 でバックフィル。

   **対象外**: Discord 由来の ID（`guild_id` / `channel_id` / `user_id` / `role_id`）は不可侵。URL／API に露出しないテーブル（`responses` / `send_log` / `assignments` / `segment_members` / `group_members` / `members`）は INTEGER／複合キーのまま。

4. **子ページ化対象**: 次の **5 つ**を子ページに移行する。

   | 子ページ | URL パス |
   |---|---|
   | 通知 新規 | `#g/<gid>/notifications/new` |
   | 通知 編集 | `#g/<gid>/notifications/<nuuid>/edit` |
   | 開催回 | `#g/<gid>/notifications/<nuuid>/occurrences` |
   | 区分メンバー | `#g/<gid>/segments/<suuid>/members` |
   | グループ分け | `#g/<gid>/notifications/<nuuid>/occurrences/<ouuid>/grouping` |

   区分の新規／編集（`segDialog`）は小フォーム（名前 + ロール紐付け）のため**モーダル維持**。これにより「フォーム＝子ページ、確認・小フォーム＝モーダル」のルールが立つ。

5. **離脱ガード**: 子ページが dirty 状態のとき、次の **6 経路**で確認ダイアログを出す。

   - SPA 内（① サイドナビ／② サーバー切替／③ 戻るボタン or ブレッドクラム）: 共通の `confirmDiscard()` を通す。
   - ブラウザ操作（④ ブラウザバック／⑤ リロード／⑥ タブ閉じる）: `beforeunload` を dirty 時のみセット。`hashchange` イベントは popstate 経由でガード判定を挟む。

   **対象外**: ⑦ アドレスバー直接編集（介入が技術的に困難なため）。

6. **保存後遷移**: 既存モーダル挙動を踏襲する。

   | 子ページ | 保存後の遷移先 |
   |---|---|
   | 通知 新規 / 編集 | 通知一覧（`#g/<gid>/`）へ戻る |
   | 区分メンバー | メンバー区分一覧（`#g/<gid>/segments`）へ戻る |
   | 開催回 | 留まる（明示の「閉じる」で戻る） |
   | グループ分け | 留まる（「保存 → Discord 投稿」フローのため） |

7. **UI 要素セット**: 子ページの標準要素を 4 点に固定する。
   - 上端: ブレッドクラム（`通知 > お知らせA > 編集` 形式、各セグメントは親 URL リンク）＋ `← 戻る` ボタン
   - サイドナビ: 親タブをハイライトしたまま
   - フォーム下部: `[キャンセル] [保存]`（既存 `nDialog` パターン踏襲）

## 根拠

- **モーダルの構造的限界**: 通知フォームと開催回画面はフィールド数が多く `.modal-lg` でも縦スクロールが深い。グループ分けはドラッグ＆ドロップに広い領域が要る。子ページにすればビューポート全体を使え、設計余地が増える。
- **ディープリンク**: テスト Worker での実機検証（ADR [0008](./0008-verify-on-deployed-test-worker.md)）や運用者間でのリンク共有時、特定の編集画面に直接飛べる価値が大きい。
- **ブラウザバック復帰**: モーダルはブラウザバックすると意図せず別画面（タブの裏側）に飛ぶ。ハッシュルーティングでブラウザ操作と SPA ナビを一致させると認知負荷が下がる。
- **シャドウキー方式の利点**: 既存マイグレーション 0001〜0011 と forward-only ポリシー（[ADR 0011](./0011-distribution-and-update-model.md)）を壊さず、テストフィクスチャ・型定義・FK 構造を温存できる。完全置換と違い「INTEGER PK の総書き換え」を回避し、URL の見た目だけを目的とした GUID 化の目的を最小コストで達成する。
- **6 テーブルに絞った理由**: `responses` / `send_log` 等は URL に露出せず内部の冪等性キーとしてのみ使う。これらに UUID を持たせる利益は無く、`send_log` のような大量行テーブルでは UUID 列追加のディスク・インデックスコストがじわじわ効く。
- **段階移行を避ける理由**: フォーム系が一部だけ子ページに移ると UI ルールが分裂し、ユーザーが「どこはモーダル、どこは子ページか」を覚えることに。一斉移行で「フォーム＝子ページ」を貫く。

## トレードオフ

- **マイグレーション 0012 の重さ**: 6 テーブルに UUID 列を追加し、既存行をバックフィルする必要がある。SQLite には UUID 関数が無いため、`lower(hex(randomblob(16)))` から 8-4-4-4-12 形式へ整形する SQL ロジックを使う。staging には grouping 検証データがあるが、マイグレーションは新規列追加 + UPDATE で完結するため stop-the-world ではない。
- **admin API パスの非互換**: 既存 `/occurrences/:id/grouping` 等の `:id` を UUID 受け取りに変更する。外部依存はなく（管理 UI 専用）影響は内に留まるが、staging で経路ごとに動作確認が要る。
- **ルータの自前実装**: 既存コードに依存ライブラリを足さず、`hashchange` + 100 行程度の文字列ルータを書く。テスト容易な代わりに、将来パターンが増えると Express ライクなライブラリが恋しくなる可能性。
- **URL 長**: グループ分けの URL は 4 段ネスト＋UUID 2 個で 120 文字程度になる。Discord embed としては問題ないが、見た目は冗長。
- **`segDialog` だけモーダルに残る不揃い**: 区分の新規／編集は小フォームのため対象外としたが、「フォームは全部子ページ」のルールから 1 件だけ外れる。将来ユーザビリティ調査で必要なら子ページ化することを残された選択肢とする。

## 実装上の補追（2026-06-27）

実装は 2 段階で着地した。

**第 1 段階（URL 連動モーダル）**: まず `ui/index.html`（単一 SPA）の DOM を保ち、子ページを `<dialog>` モーダルとして URL 連動で開閉する形にした。`navigate()` がハッシュを更新 → `hashchange` → `handleHashRoute()` が対応モーダルを開く。この時点でディープリンク・ブラウザバック・離脱ガード（決定 #2/#5）は満たしたが、「メインエリアのみ差し替え（決定 #1）」は未達でモーダルが浮いていた。

**第 2 段階（本物のページ遷移＝決定 #1 達成）**: 子ページの 4 つの `<dialog>`（`nDialog`/`occDialog`/`groupingDialog`/`segMembersDialog`）を `#workspace main` 直下へ移設し、開く際に `showModal()` ではなく **`show()`（非モーダル＝背景暗転なし・top-layer でない）** を使い、`main.has-page > section { display:none }` でリスト（タブの一覧）を隠して**メインエリアを占有**させる。これで「フォーム＝子ページ、ヘッダ／サイドナビは常駐」を満たす。各ページ上端にパンくず＋`← 戻る`（決定 #7）を付けた。`segDialog`（区分の小フォーム）はモーダル維持（決定 #4）。

実装メモ:
- ページ開閉は `openAsPage()`/`closeAsPage()`/`syncHasPage()` に集約。`has-page` は **ワークスペース側 `<main>`**（ピッカー側 `<main>` と別物）に限定して付与する。
- サイドナビ切替は開いている子ページを離脱ガードののち閉じ、URL を base に同期する。
- `handleHashRoute` の `grouping` 分岐は親の `occDialog` も閉じてから開く（多重表示の防止）。

UUID シャドウキー（決定 #3）と API パスの UUID 化、3 層シェルのページ遷移（決定 #1）は実装で完全に実現済み。preview + `window.api` スタブで、各ボタン → ページ遷移（`main.has-page`／一覧 `display:none`／非モーダル）、戻る・サイドナビでの一覧復帰、URL 同期・ブラウザバックを検証した。

- ADR [0007](./0007-admin-ui-redesign.md) の決定 #1（2 層シェル）。サーバー選択 → 左サイドナビの構造は維持し、その上に子ページ層を追加する形で拡張する。
- ADR [0007](./0007-admin-ui-redesign.md) の決定 #2（Snowflake 手打ち全廃）と決定 #3（繰り返し UI の平易化）は影響を受けない。
