# 通知配信パイプライン刷新（通知ごと送信時刻・高頻度cron・ペース配信・送信ログ）

通知の cron 配信を「1日1回固定 21:00・無記録・無制御」から、「**通知ごとに送信時刻を持ち、毎分 cron ＋送信予算でペース配信し、全送信を記録（冪等台帳兼可視化）する**」パイプラインへ刷新する。機能 ③（送信時刻）・④（送信の可視化）・⑦（送信上限＝エラー対策）を 1 つのサブシステムとして扱う。用語は [CONTEXT.md](../CONTEXT.md) の「送信時刻」、回答締切連携は [ADR 0014](./0014-response-deadline-and-change-detection.md) を参照。

## 文脈

- 現状 cron は `wrangler.jsonc` 固定 `0 12 * * *`（JST 21:00・1日1回）。**運用者が送信時刻を変えられない**（③要望）。
- リマインド DM は BOT と受信者しか見えず、**主催が「ちゃんと送られたか」を確認できない**（④要望）。ただし未回答者を全体に晒し上げるのは不可。
- Cloudflare Free の律速は **subrequest 50／1実行**（= Discord API 呼び出し回数。D1 はバインディング経由で subrequest に数えない）。多区分 × 多通知 × 多人数だと **1実行で 51件目がエラー**（⑦懸念）。cron 実行回数（毎分でも 1,440/日）・日次 100,000 invocation は誤差。
- 現実的 worst case ＝ **10区分 × 10通知 × 100人 = 10,000 DM/日**。Free ハード上限 `1,440実行 × 50 = 72,000/日` に対し約 7 倍の余裕があるが、**無制御だと 1実行 50 超過や Discord anti-spam リスク**が残る。初回接触は「DMチャンネル作成＋送信」で 2 call になる点も考慮が要る。

## 決定

- **通知ごと送信時刻 `send_hour`（時単位・既定 21）**：cron 駆動の全送信（募集／未回答・未定リマインド／ノルマ／締切告知）を、その通知の `send_hour`（JST）に送る。開催の `start_time` とは**別物**（送る時刻 ≠ 開催の時刻）。未設定（既存）は 21 でバックフィルし現行挙動を維持。
- **cron を毎分（`* * * * *`）に変更**：毎分を「**ドレインの時計**」とし、各ティックで「送信時刻に達し、かつ未送信の due な送信」を**予算分だけ**送信し、残りは次ティックへ繰り越す。
- **送信予算（per-tick budget）＝設定値**：Free 既定 **45**（subrequest 50 律速の安全側）。**Paid 化で引き上げ可能**（毎分 cron の実時間律速で ~120/実行 → 容量 ~172,800/日）。`config` テーブルのキーで保持し、**課金後に上限を突破できる**ようコード分岐なしで運用者が調整する。
- **送信ログ `send_log`（新テーブル）を「冪等台帳 兼 ペースカーソル 兼 可視化」に兼用**：1送信ごとに `(通知 / 開催回 / 宛先 / 種別 / 送信日 / 成否 / 失敗理由)` を記録。`UNIQUE(notification_id, occurrence_id, user_id, kind, send_date)` で **1日1回・二重送信防止**。④の可視化（主催が送信状況・失敗＝DM拒否等を**完全非公開**に確認）と ⑦のカーソルは同一構造なので **1テーブルで足りる**。
- **推奨上限の提示（ハード強制しない）**：管理 UI に「**現構成の推定ピーク送信 vs 推奨上限**」を表示し、超過を警告する。Free 推奨 ＝ **総 ≤ 10,000/日 ・ 同一送信時刻の集中 ≤ 2,500**（毎分 cron・予算 45 で約 55 分以内に流し切れる量）。Paid で予算を上げれば推奨上限も自動で緩和。
- **DM 個別配信は維持**：チャンネル `@ロール` 一斉配信モードは**作らない**（現機能構成を変えない方針）。「大規模はチャンネル配信が桁違いに軽い（200人 → 1 call）」点は**文書で案内するに留める**。
- **送信時刻＝負荷分散装置**：通知ごとに時刻を散らせば 1 窓あたりの送信が割れ、Free でも捌きやすくなる。同一時刻への集中は UI 警告で抑止する。

## 根拠

- subrequest 50/実行が真の律速なので、**予算でティックを 50 未満に保てば構造的にエラーが発生しない**。cron 回数・日次 invocation は枠に対し誤差。
- 毎分 cron ＋冪等台帳により「**指定時刻に・取りこぼしなく・二重送信なく**」流せる。④の可視化ログと ⑦のペースカーソルが同一構造のため、1 テーブルで両方を賄える。
- 予算を設定値にすることで「**Free は安全・Paid は突破**」を分岐なしで実現（運用者の knob）。
- worst case 10,000/日が Free 余裕約 7 倍に収まること、初回接触 2 倍の最悪日（20,000）でも 72,000 未満であることを実数検証済み。

## トレードオフ

- 毎分 cron は idle ティックが多い（1,440/日）。各ティックは「due があるか」を軽い 1 クエリで確認する no-op で、100k/日 invocation 枠の約 1.4%。
- ペース配信はカーソル／原子的記録が要る分、daily 一括より実装が複雑。ティック競合・リトライに備え送信記録は**原子的（UNIQUE 制約に依拠）**に行う。
- DM 維持のため、**超大規模（数万/日超）は Free では不可・Paid 前提**。チャンネル配信モードを入れない判断で、超大規模の軽量化は将来課題として残す。
- 推奨上限はソフト（警告のみ）。運用者が無視して大量設定すると、ペース配信が流し切るのに時間がかかる（タイムリー性低下）／Paid 推奨に倒れる。
- Discord 側 anti-spam は Cloudflare とは別の天井。大量 DM は bot 制限リスクがあるため、推奨上限はその配慮も兼ねる。

## スキーマ影響（migration 連番は実装時に確定・現最新 0009）

- `notifications.send_hour INTEGER NOT NULL DEFAULT 21`（JST 時・0〜23）。
- 新テーブル `send_log`（`id` / `notification_id` / `occurrence_id`（null可）/ `user_id`（null可・チャンネル投稿時）/ `kind`（`recruit`|`remind_unanswered`|`remind_undecided`|`quota`|`deadline_notice`|`change_alert`）/ `send_date`（JST 'YYYY/MM/DD'）/ `status`（`sent`|`failed`）/ `error`（null可）/ `created_at`）＋ `UNIQUE(notification_id, occurrence_id, user_id, kind, send_date)` ＋ 閲覧用インデックス。
- `config`（既存 key/value）に `send_budget_per_tick`（既定 45）。
- `wrangler.jsonc` の `crons` を `0 12 * * *` → `* * * * *`（配布 fork にも Sync fork で伝播）。
