# 回答締切と締切後の変更検知・通知・可視化

Occurrence に任意の「**回答締切**」を設け、締切後の Response 変更を検知して管理者へ**チャンネル通知**し、回答履歴で「締切後変更か否か」を識別できるようにする（機能①）。出欠/勤怠運用で「**確定後にこっそり変えられる**」のを運用者が把握するための仕組み。用語は [CONTEXT.md](../CONTEXT.md) の「回答締切（確定日時）」、配信・告知の冪等化は [ADR 0013](./0013-notification-delivery-pipeline.md) を参照。

## 文脈

- 現状、Response（参加/不参加/未定）は**いつでも変更でき**、締切やロックの概念は無い。単発の `decided_occurrence_id`（候補日確定）は**別概念**で、混同しないよう今回のものは「回答締切」と呼ぶ。
- 出欠/勤怠運用では「ロスター確定後の変更」を主催が把握したい。ただし**ボタンの固定（ハードロック）はしたくない**（変更自体は許容し、把握だけしたい）。
- 変更履歴の監査テーブルは無い（`responses.updated_at` のみ）。

## 決定

- **回答締切（任意・通知ごと）**：`response_deadline_hours`（開始の **N 時間前**・null＝締切なし）。その通知の**全 Occurrence に自動適用**する。
- **ソフトロック**：締切後も回答ボタンは固定しない（Response の変更は技術的に可能なまま）。
- **締切告知（メンバー向け・2 段）**：
  1. 募集投稿に「**回答締切: M/D HH:MM**」を日時行と同様に**自動表示**。
  2. 締切後の最初の cron ティックで「⏰ 締め切りました／以降の変更は主催者に記録・通知されます」を**募集チャンネルへ追従投稿**（送信ログ `kind=deadline_notice` で冪等化・[ADR 0013](./0013-notification-delivery-pipeline.md)）。
- **締切後変更の検知＝押下時に即時**：interaction でボタン押下時、**既存回答（旧）と新を比較**し、現在が締切後で**差異がある**（**未回答 → 回答の初回も含む**）なら検知イベントとして扱う。押下時に旧値が読めるため**監査テーブルは不要**。
- **変更通知（管理者向け）＝チャンネル投稿のみ・メンションなし**：`change_alert_channel_id`（null 時は通知の `channel_id`）へ「**＠変更者 が [旧] → [新] に変更（開催 M/D）**」を**メンション無し**で投稿する。可視範囲はそのチャンネルの権限で運用者が制御する（晒し上げ・過剰通知の回避）。
- **回答履歴で識別**：`responses.post_deadline_change`（0/1）を立て、回答履歴（機能②）に「**締切後変更**」列で表示する。
- **回答不要（announce-only）通知は対象外**：Response を集めないため締切も無意味。UI でも締切欄を隠す。

## 根拠

- ソフトロック＋検知＋通知は「**変更は許すが把握する**」という運用要求に正確に対応する。ハードロックは現場の正当な例外変更まで阻害する。
- 押下時即時検知なら**監査テーブル不要・遅延ゼロ**で旧→新を出せる。
- 同一ユーザが同じステータスで連打しても `status !== oldStatus` のため**初回のみ**フラグが立ち、2 回目以降は発火しない（同値再押下で誤通知しない）。
- **無メンションのチャンネル投稿**は、管理者が見る場所に静かに残せて晒し上げにならない。「誰に見せるか」はチャンネル権限で運用者が決める。

## トレードオフ

- 締切告知の追従投稿（②）は cron 駆動のため締切ちょうどではなく次ティック（毎分 cron 化済み＝最大 1 分程度の遅れ）。
- 変更通知が**無メンション**のため、管理者がそのチャンネルを見ていないと気づかない（プッシュが要るなら将来メンション可に拡張余地）。
- `change_alert_channel_id` 未設定時は募集チャンネルへ出るためメンバーからも見える。**専用チャンネルの設定を推奨**。
- 締切後変更フラグは「締切後に変えたか否か」の**真偽のみ**（変更回数・変更履歴は保持しない）。詳細監査が要れば将来 `send_log` の `change_alert` 行で代替できる。

## スキーマ影響（migration 連番は実装時に確定）

- `notifications.response_deadline_hours INTEGER`（null 可）
- `notifications.change_alert_channel_id TEXT`（null 可）
- `responses.post_deadline_change INTEGER NOT NULL DEFAULT 0`
