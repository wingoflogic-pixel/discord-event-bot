# 番号割り当てを 2 モード（早い順 / 完全ランダム）に変更し、安定割り当て原則を破棄する

ADR [0015](./0015-grouping-and-pair-constraints.md) と当初の実装は、Assignment（番号割り当て）について「Fisher–Yates でランダム」かつ「既存番号は維持し新規参加者にのみ空き番号を振る」**安定割り当て**を採用していた。**この方針を改め、運用者が「早い順」「完全ランダム」のどちらかをボタンで選び、毎回その回の番号を全て振り直す方式に切り替える**。

## 文脈

- 実運用で「**回答が早かった人に若い番号を渡したい**」という要望が出てきた（イベント現場で「先に手を挙げた人ほどステージに近い席を取る」等の運用に直結）。安定割り当てではこの欲求を満たせない。
- 一方で「完全ランダム」にも需要は残る（プライズ抽選など、運用者が「事前順序を一切持たない」ことを担保したい場面）。両者は二者択一ではなく、**運用者がその回の意図に応じて選ぶ**設計が適切。
- 「既存番号維持」は本来「既に告知済みの番号を変えると現場が混乱する」事故防止のための制約。しかし運用者が **明示的に「振り直す」ボタンを押す** モデルなら、振り直しは「意図された運用」であり事故ではない。安定割り当ての価値は十分高くない。
- 新スキーマ列は不要。`responses.updated_at` が「その人が現在の status に最後に変わった時刻」を保持しているので、`status='参加'` で絞れば「最後に参加表明した時刻」として利用できる。

## 決定

### 2 モード

- **早い順（`first-come`）**：`responses.updated_at ASC, responses.user_id ASC` で並べ、1..N を順に振る。
- **完全ランダム（`random`）**：参加者を Fisher–Yates でシャッフルし、1..N を順に振る。
- どちらのモードでも対象は `status='参加'` の Member のみ。不参加・未定・未回答は対象外（既存仕様維持）。

### 全振り直し（安定割り当て破棄）

- どちらのモードでも、実行時にその Occurrence の `assignments` を **全削除** してから 1..N を新規 INSERT する。
- 結果として、毎回番号は変わりうる。運用者は「振り直す」ことを明示的に選んで実行する。

### API

- `POST /occurrences/:uuid/assign` のリクエストボディに `{ mode: 'first-come' | 'random' }` を必須で受ける。未指定 / 不正値は `400`。
- レスポンス形式（`{ assigned, all }`）は維持。ただし `assigned` は今回振り直した分（= `all` と同じ集合）になる。

### UI

- 開催回画面の「番号割り当てを実行」1 ボタンを、「早い順で割り当て」「ランダムで割り当て」の **2 ボタン** に分割する。
- 各ボタンの confirm ダイアログで「既存の番号は破棄され、新しく振り直されます」を明示。

### 同秒タイブレーク

- 早い順で `updated_at` が同一の場合、`user_id` 昇順で決定論的にタイブレーク。`Math.random()` 系の非決定要素は使わない（テスト容易性 + 再現性のため）。

### 「最後に参加に戻した人」の扱い

- `responses.upsertResponse` は ON CONFLICT で `updated_at = excluded.updated_at` に上書きする（[src/db/responses.ts:19-30](../../../src/db/responses.ts#L19-L30)）。つまり「未定→参加」「参加→不参加→参加」と動いた人の `updated_at` は **最後に "参加" に変わった時刻** になる。
- そのため、参加表明を一度撤回して戻した人は **その回の中で後発扱い** になり、`first-come` モードでは大きい番号を取る。これは仕様として受け入れる（運用者が早い順を選んだ意図と整合）。

## 根拠

- **「早い順」運用要望への直球解**：新スキーマなしで実現できる。`updated_at` の意味（「現 status に最後に変わった時刻」）が「最後の参加押下時刻」と一致するため。
- **モード選択制**：「ランダム」「早い順」を同列のオプションとし、運用者の意図を UI レベルで吸収する。隠れたデフォルト挙動に依存しない。
- **全振り直しに統一**：安定割り当ては「2 つのモードを併存させる」とロジック分岐が深くなる（mode×安定維持の 4 組合せ）。毎回振り直しに一本化することで実装も挙動も単純化される。
- **明示ボタンによる事故防止**：「振り直したくない」事故への対策は UI（confirm 文言）と運用ルールに寄せる。バックエンドの「既存維持原則」を残すと、運用者が「ボタンを押したのに番号が変わらない」と混乱しやすい。

## トレードオフ

- **既存番号が動きうる**：運用者が「振り直し」を押すと番号が変わる。既に告知済みの番号がある場合は再告知が必要。これは「振り直しボタン」という UI を明示することで運用者の責任に倒す。
- **欠番回収の議論が消える**：既存の「欠番優先」ロジックは廃止。再実行のたびに 1..N の連番に揃うので、欠番自体が発生しない（不参加に変えた人がいても、次回実行で残り参加者だけで詰め直る）。これは利点でもあり、「事前番号と次回番号の対応が変わる」点では運用者教育が必要。
- **冪等性の喪失**：同じ参加者で 2 回連続実行すると `random` モードでは番号順序が変わる。テストは「集合一致」「1..N 連番」など順序非依存な観点でアサートする必要がある。
- **過去 ADR との不整合**：ADR 0015 と CONTEXT.md の「安定割り当て」記述を改訂する。本 ADR が直接の根拠（override）。

## スキーマ影響

- なし（新規列・新規テーブル・新規 migration とも不要）。
- `assignments` テーブル定義は変更しない。
