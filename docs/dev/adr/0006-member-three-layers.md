# メンバーは 3 層で扱う（身元＝グローバル／在籍＝ライブ取得／対象＝区分）

メンバーを 1 枚の名簿として持たず、**性質の異なる 3 層**に分けて扱う。マルチサーバー（ADR [0004](./0004-multi-server.md)）で「同一 Discord ユーザーが複数サーバーに現れる」ことを正しく扱うための整理。

## 決定

- **① 身元（identity）＝ グローバル**：`members`（`user_id`・`dm_channel_id` 等）は全サーバー横断で 1 行。Discord アカウントと DM チャンネルは本質的に横断一意なため、二重持ちしない。
- **② 在籍（roster）＝ ライブ取得・非永続**：「そのサーバーにいる人」は Discord API（`GET /guilds/:id/members`）から都度取得し、DB に同期テーブルを持たない。
- **③ 出欠対象（audience）＝ 区分所属**：ポーリング・リマインド・ノルマ・集計の母集団は **メンバー区分（Segment）のアクティブ所属者**のみ。在籍していても区分に入れなければ対象外。
- ローカルの `members` 行は、**区分への追加、または募集ボタン回答の時点で自動生成**する（User ID 手打ちを廃止）。追加時に **Discord のサーバー内ニックネーム**も取得して表示名に用いる。

## 根拠

- 実行時に必要なのは ③ だけ。② は「区分に追加する時の選択元」と「名前表示」にしか使わないので、全員を永続同期するのは過剰でドリフトを生む。
- ライブ取得のメンバーピッカーにより「マスタ＝サーバー全員」をユーザー体験として満たしつつ、DB には関係者だけが残る。
- サーバー内ニックネームを取得することで「表示名がグローバル 1 個」だった不正確さも解消する。

## トレードオフ

- メンバー一覧・ニック取得に **Discord「Server Members Intent」（特権インテント）の有効化が必須**。前提として bot がそのサーバーに参加している必要があり、未有効時の応答は環境により **403 もしくは空配列**（実装時に要検証）。セットアップ手順に必須項目として追記する。
- `GET /guilds/:id/members/search` は特権インテント不要だが**部分一致補助**であり、全員列挙の代替にはならない。
- 特権インテントは **bot が 100 サーバーを超えると Discord の審査対象**になる（将来制約・現時点は非目標）。
- `display_name` はグローバル 1 行のため、**複数サーバーに在籍する同一 Member の表示名は、最後に取得したサーバーのニックで上書きされうる**（決定①の身元グローバルと③の区分＝サーバー局所の張力）。実害は小さく許容するが、将来は所属（区分）側に表示名を持たせる選択肢を残す。
- 区分に未所属の在籍者は DB に永続しない（実行時に不要なので問題なし）。
