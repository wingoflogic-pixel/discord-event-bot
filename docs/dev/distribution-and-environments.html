配信と利用環境ガイド

「開発者のテスト環境（staging）」「開発者の本番環境（Choiemu）」「ユーザーの本番環境」の3つがどう違い・どうつながるのか、ユーザーがどうセットアップするのか、データベース（D1）はどう立つのかを、図とともに要点解説する文書です。3環境はいずれも「公開GitHub → Cloudflare Workers Builds」で動きます（ADR 0012）。出典は本リポジトリの設定ファイル・ADR と Cloudflare 公式ドキュメント（末尾に一覧）。Cloudflare の挙動に関する記述は公式ドキュメント・GitHub で確認したものですが、配布の通し挙動の最終確認は実デプロイで行う前提です。

この文書の更新状況（2026-06-20）

ユーザーの初回セットアップ手段を 「Deploy to Cloudflare ボタン」から「公開リポジトリを Fork → Cloudflare Workers Builds に接続」へ変更しました（案B・ADR 0011 追補）。理由は、Deploy ボタンが作るのは fork ではなく clone で、更新の「Sync fork」が使えないためです。本文の §2・§3 は旧（Deploy ボタン前提）の記述が残っており、別途改訂予定（タスク B-6）です。最新の正は setup.html・ADR 0011 追補 を参照してください。

1. 全体像：開発と3つの環境

登場するのは 開発者のローカル（開発の起点） と、そこから公開GitHub経由でデプロイされる3つの環境です ── ② 開発者のテスト環境（staging）／③ 開発者の本番環境（Choiemu）／④ ユーザーの本番環境。ADR 0012 の方針では、この3環境はすべて同じ 「公開GitHub → Cloudflare Workers Builds」 経路で動きます（＝デプロイ機構が同一＝環境差が出ない）。違いは「どのブランチ/forkを見るか」と「どのDiscord・データに繋がるか」だけです。

  • 開発者の本番（③）とテスト（②）は分離。 同じ機構でも、③は本番Choiemu＋本番D1、②はテスト用Discord＋テストD1に繋がる別環境（データは完全に独立）。
  • 3環境ともデプロイ機構は同一（parity）。 公開GitHub → Workers Builds が npm run deploy（マイグレーション＋デプロイ）を自動実行する（ADR 0008／ADR 0012）。だから②で本番と同条件の検証ができる。
  • インスタンスは環境ごとに別物。 それぞれ別のWorker・別のD1で、データは独立（ユーザー環境はテナント分離ではなく各自が自己ホスト）。
  • 移行中。 現状は③開発者の本番のみローカルCLI（npm run deploy:cli）で稼働、②staging は未構築。Workers Builds 接続後に上図へ統一する（§5）。

2. 3つの環境（開発者のテスト・開発者の本番・ユーザー本番）

3つの環境はすべて 「公開GitHub → Workers Builds」 という同じ仕組みで動きます（＝デプロイ機構が同一＝parity。ADR 0012）。開発者のソースの正本はローカル（PC）にあり、それを公開GitHubへ push して全環境の起点にします。各環境の違いは 「どのブランチ/forkを見るか」（②=staging／③=main／④=利用者の fork）と、「どのDiscord・データに繋がるか」だけです。とりわけ ② 開発者のテスト環境と ③ 開発者の本番環境（Choiemu）は、同じ機構でもデータ・接続先が完全に独立した別環境として分離します。

3. ユーザーのセットアップとD1の自動構築

ユーザーが必要とするのは無料の3アカウントだけ。データベース（D1）はユーザーが手で作る場面が一切なく、Deployボタンが自動生成し、スキーマもマイグレーションで自動適用されます。

必要なアカウント用途費用
DiscordBotアプリ＋導入先サーバーの管理権限無料
Cloudflareホスティング（Worker）＋データベース（D1）無料枠
GitHubDeployボタンが作る fork の置き場＋更新（Sync fork）無料

D1がゼロ操作で立つ仕組み（Q への答え）

Deployボタンが wrangler 設定のD1バインディング宣言を読み、ユーザーのアカウントにD1を自動生成します（後述リスク①のとおり、自動生成には database_id を記載しない配布用設定が前提）。デプロイ時に deploy スクリプトが先に wrangler deploy でD1を自動生成して id を wrangler.jsonc に書き戻し、その後 wrangler d1 migrations apply DB --remote がその id を解決して migrations/ の全連番（v7.0.1 時点で 0001〜0009）を順に適用しスキーマを自動構築します。マイグレーションを「DB名」でなくバインディング名 DB で指定しているのは、ユーザーごとにD1名が変わっても同じスクリプトで動かすためです。なお「合言葉」とは Deployボタンの入力欄で渡す4つのシークレット（DISCORD_PUBLIC_KEY / DISCORD_APPLICATION_ID / DISCORD_BOT_TOKEN / ADMIN_TOKEN。出典 .dev.vars.example ／ package.json の cloudflare.bindings）のことです。

「ローカル→Cloudflare」でGitHubを省けないのか？

省けますが、GitHub登録（Webフォーム1回）の代わりにCLI環境一式（Node＋wrangler＋ターミナル操作）を背負わせることになり、非エンジニアには逆に難しくなります。Cloudflareで「Worker＋D1＋マイグレーション」をノンCLIで立てる経路は実質Deployボタンだけで、それは公開gitリポジトリを参照するため、GitHub（またはGitLab）は原理的に外せません。さらにGitHubは「更新がSync fork 1回」も成立させているので、継続的にも効きます。結論：3サービス（GitHub/Cloudflare/Discord）が最小かつ最簡。

4. リスク・要対応

公開配布を本当に成立させる前に、以下を確認・対応する必要があります。重大度は確度×影響で付与（最優先＝放置すると配布が成立しない／中＝品質・信頼性の問題／対応済み＝本変更で修正）。①③は本変更で対応済み（①は通し動作の最終確認のみ実デプロイで残る）、②（Workers Builds のデプロイコマンド設定）は Cloudflare ダッシュボードでの設定が要る未対応項目です。

対応済み ① database_id のベタ書き（本変更で対応・実デプロイ確認のみ残る）

事実（旧）旧 wrangler.toml に開発者のアカウントの実 database_id（f6ff753c-…）がベタ書きされていた。
影響Cloudflareの自動プロビジョニングは「IDが書かれているバインディング＝既存リソース」とみなして新規作成をスキップする（Cloudflare公式・末尾出典で確認）。そのIDはユーザーのアカウントに存在しないため、放置すればユーザーのデプロイが失敗する蓋然性が高い。
対応（実施済み）本変更で対応済み: ①設定を wrangler.toml → wrangler.jsonc へ移行し（自動生成IDの書き戻しが .toml で行われない issue #13632 を回避）、②database_id を記載しない（省略する）ことで自動プロビジョニングを発火させ、③package.json の wrangler 下限を ^4.102.0（PR #14275 初収録版）に引き上げた（id 省略でもバインディング名 DB からUUIDをAPI解決して wrangler d1 migrations apply DB --remote が通る）。lockfile は解決版 4.103.0 のまま・スペック文字列のみ手修正し、@emnapi ピン留め（Deployビルド対策）と npm ci 整合を維持。ローカル全87テストが通過することを確認済み。
保守者の落とし穴と対応同じ database_id 省略設定をローカルでも使うと、開発者のローカル wrangler deploy も自動プロビジョニングで別のD1を作り本番Choiemuデータから切り離される。対応として、実 database_id を持つ wrangler.local.jsonc（gitignore済み・配布しない）を --config 指定する npm run deploy:cli を追加した。素の npm run deploy は Workers Builds／Deployボタン用、本番への CLI デプロイは deploy:cli を使う。これは §5 の本番 Workers Builds 移行までの移行措置で、移行後は撤去する。
残課題Deployボタン配布時の「D1自動生成＋マイグレーション自動適用＋デプロイコマンド」の通し動作の最終確認は、§5 の②検証環境（Workers Builds）または初回実デプロイで行う（ADR 0011 が「未検証」とする項目）。本ガイドはこの通し検証を成立させる前提として database_id 空化・jsonc 化・floor 引き上げを行った。

最優先 ② Workers Builds のデプロイコマンド設定

ADR 0011・dev-and-release-flow.md は「Workers Builds は本番ブランチへの push ごとに deploy コマンドを実行する」前提に立っていますが、その既定コマンドは npx wrangler deploy（マイグレーションを実行しない）です（Cloudflare公式・末尾出典）。本プロジェクトはマイグレーション込みの npm run deploy（package.json:9）を走らせる必要があるため、Workers Builds のビルド設定でデプロイコマンドを npm run deploy に上書きしなければなりません。未設定だと Sync fork してもスキーマ更新が走らず不整合になります（確定事実なので①と同格・最優先）。これは②開発者のテスト環境・③開発者の本番環境・④ユーザー本番のすべてで必要な設定です。

対応済み ③ setup.html の「用意するもの」にGitHubが無い（修正済み）

「用意するもの」が Discord・Cloudflare の2つだけで GitHub を欠いていた（同ファイルの更新手順では fork の Sync fork を要求しており自己矛盾）。本変更で setup.html・README.md・docs/dev/dev-and-release-flow.md・.claude/rules/dev-and-release.md を「Discord・Cloudflare・GitHub の3つ」に統一修正済み（GitHub＝fork作成と更新に使う・無料）。

5. 採用方針：3層構成で dev/prod parity（ADR 0012）

この方針は採用済み（2026-06-20）／実装は段階移行中

配布用設定の database_id 空化・wrangler.jsonc 化・wrangler 下限引き上げ・deploy:cli 移行措置は導入済み（§4①）。残りは staging／本番を Workers Builds に接続する Cloudflare ダッシュボード作業で、本番が Workers Builds に乗り次第 deploy:cli は撤去します。決定の根拠と却下案は ADR 0012 に記録。

§1・§2 との対応。 ここでの3層は開発者側の開発パイプラインの視点で、① ローカル開発＝§1/§2 の「開発者のローカル」、② 検証(staging)＝「開発者のテスト環境」、③ 本番(prod)＝「開発者の本番環境（Choiemu）」に対応します。ユーザーの本番環境（④）は③と同じ Workers Builds 機構を、利用者の fork で動かしたものです。

動機（なぜ必要か）。 これまでは②検証＝ローカルCLI（wrangler deploy）／④ユーザー本番＝Workers Builds とデプロイ機構が分かれていました。機構が違うと、その差に起因するエラー——自動プロビジョニング（空 database_id からのD1生成）・Workers Builds 上でのマイグレーション実行・デプロイコマンド設定——が検証をすり抜けて本番で初めて出ます。そこで「開発者の本番も、ユーザーと同じ GitHub→Cloudflare（Workers Builds） 経路で動かす」。さらに②検証も同じ Workers Builds 経路にして、検証と本番のデプロイ機構を完全に一致させます。

なぜ良いか（4つの効果）。

  • 環境差由来のエラーを②で先に潰せる。「Workers Builds が npm run deploy を実行 → D1自動生成 → マイグレーション自動適用」という一連が②で本番と同条件で回るため。
  • database_id 空化が②で実証される。ADR 0011 唯一の「未検証」項目（配布時のD1別名生成＋マイグレーション自動適用の通し動作）が、本番に触れずに検証される。
  • ローカルからの本番リモートデプロイ（毎回ユーザー許可が要る本番操作＝deploy:cli）が不要になる。リモートへのデプロイが全部 Workers Builds 経由になり、運用が簡素化される。
  • ADR 0008 の安全目的をより完全に満たす。Workers Builds は Cloudflare CI 上で走りシークレットはダッシュボード管理（ローカル .env を読まない）なので、ADR 0008 が恐れた「本番値混入による誤投稿」の経路自体が消える。

現在のローカルからのリモートデプロイ（npm run deploy:cli）は最終的に廃止され、ローカルは論理検証（vitest／sealed wrangler dev）専用になります。

具体的な形

  • ② 検証(staging) … staging ブランチ（または検証用fork）に専用の staging Worker を Workers Builds で接続。テスト用Discordアプリ＋自動生成D1＋テストデータに向ける。
  • ③ 本番(prod) … main（または本番fork）を本番 Worker に Workers Builds で接続。実Choiemu＋他イベントは1インスタンスのマルチサーバー（guild_id・ADR 0004）で共存。
  • ②③とも デプロイコマンドを npm run deploy に設定 → 機構は同一。Workers Builds の既定は npx wrangler deploy（マイグレーションを実行しない）なので必ず上書きが必要（リスク②）。違いは接続先Discordアプリとデータだけ。
  • ① ローカル開発は残す（降格するのは「ローカルからのCLIリモートデプロイ経路」＝deploy:cli だけ。ローカルの開発・テストはむしろ速い反復の主役）。

昇格パイプライン： ①ローカルで実装・npm test → staging へ push → ②で Workers Builds が自動デプロイ・検証 → main へ昇格（③へ反映）。

正直な注意点

  • フィードバックが遅くなる（push→ビルド→デプロイ）。だから①ローカル（vitest／sealed wrangler dev）は残し、速い反復は①、②は最終パリティ確認に限定する。3層はそれぞれ役割が違う（①速さ・②忠実さ・③本番）。
  • ADR 0008 の改定が必要（「検証はデプロイ済みテスト機で」の精神は維持しつつ手段をCLI→Workers Buildsへ）。本変更で ADR 0008 に追補済み。
  • 初期構築コスト。staging Worker＋ブランチ＋Workers Builds設定（デプロイコマンドの上書きを含む）＋②用シークレットのダッシュボード投入が一度必要。
  • 依存順序。リスク①（database_id 空化＆jsonc 化）は②を成立させる前提として先に必要で、これは本変更で対応済み。
  • 移行期の二重管理。本番が Workers Builds に乗るまでは wrangler.local.jsonc／deploy:cli を維持（wrangler.jsonc 変更時は追従させる。--config は設定を置き換えるため内容がドリフトしうる）。

守るべき規律：「②で検証 → main へ昇格（③へ反映）」「DB変更ありの前にD1バックアップ」「③が追うブランチは安定版に保つ」。複数イベント（Choiemu・他）はマルチサーバーで共存でき、イベントごとにデプロイを分ける必要はありません。
